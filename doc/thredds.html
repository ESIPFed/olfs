THREDDS Catalogs
Server4 (version: beta-1.4)

 

This release of Server4 supports the complete THREDDS catalog service stack. THREDDS catalogs are controlled by a catalog.xml file located in the (persistent) content directory for the OLFS (More on that here). Rather than provide an exhaustive explanation of the THREDDS catalog functionality and configuration I will appeal to the existing documents provided by our fine colleagues at UNIDATA:

  • Catalog Basics
  • Catalog Specification
  • Catalog Primer
  • DatasetScan Element

Did you read all that? Excellent...

READ THIS NEXT BIT OR ABANDON ALL HOPE!

Now, to get THREDDS catalogs working on Server4 YOU MUST do at least the following:

  1. For each collection that appears in the top level of the OPeNDAP directory response (http://localhost:8080/opendap/contents.html if Server4 is running out of the box on this machine) you MUST create a <datasetScan> in the catalog.xml file located in the persistent content directory ($TOMCAT_HOME/content/opendap).

    The THREDDS catalog views will NOT include a collection for which this is not done!

  2. In each <datasetScan> element that you create you MUST use the following element: <crawlableDatasetImpl className="opendap.olfs.BESCrawlableDataset" /> This is the only CrawlableDataset implementation available!

  3. No matter what your BES collection looks like, the location attribute of the <datasetScan> element must begin with "/root". This is a THREDDS requirement.

  4. The service attribute in the <datasetScan> element must be set to "OPeNDAP".

  5. The path attribute in the <datasetScan> element appears in the URL after the servlet name, and MUST be the same as the value of the location attribute with the leading "/root/" removed. In other words it MUST NOT start with a "/" character .

  6. You should apply a filter to the data that coincides with the value of the "BES.Catalog.catalog.TypeMatch" for the data types being served. I suggest that you make the filter expose ALL of the data types served by the BES. See the THREDDS pages on the DatasetScan Element for filter details. The point of this to remove files from the catalog view that are NOT OPeNDAP data. For example README files.

  7. Add metadata as you see fit. If you REALLY did read the THREDDS documentation above you will already have a clue about this. If you haven't DO SO NOW.

 

Examples

Example 1

Here is an example catalog.xml file for a Server4 installation in which the top level of the BES shows only ONE collection called "data":

<?xml version="1.0" encoding="UTF-8"?>
<catalog name="Server4 Test Catalog"
         xmlns="http://www.unidata.ucar.edu/namespaces/thredds/InvCatalog/v1.0"
         xmlns:xlink="http://www.w3.org/1999/xlink">

     <!-- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -->

     <service name="OPeNDAP-Server4" serviceType="OPeNDAP" base="/opendap/"/>

     <datasetScan location="/root/data" path="data" name="AllMyData" serviceName="OPeNDAP-Server4">

	       <crawlableDatasetImpl className="opendap.olfs.BESCrawlableDataset" />

           <filter>
               <exclude wildcard=".*" atomic="true" collection="true" />
               <include wildcard="*" />
           </filter>
           <addDatasetSize />

           <metadata inherited="true">
               <serviceName>OPeNDAP-Server4</serviceName>
               <authority>opendap.org</authority>
               <dataType>Random</dataType>
           </metadata>
     </datasetScan>

     <!-- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -->
</catalog>

In particular note the following:

  • The line in which the CrawlableDataset implementation is defined: 
    <crawlableDatasetImpl className="opendap.olfs.BESCrawlableDataset" />
    Identifies the correct CrawlableDataset class for Server4 - the one that works with the BES to automatically generate catalogs.


  • In the <datasetScan> element the location attribute's value MUST begin with /root. So, if the top level collection in the BES contains 4 sub collections they may each be identified using a separate <datasetScan> element like so: 
    	<datasetScan location="/root/nc"  path="nc"  name="NetCDF Archive"   serviceName="OPeNDAP-Server4"> . . . </datasetScan>
	<datasetScan location="/root/hdf" path="hdf" name="HDF Archive"      serviceName="OPeNDAP-Server4"> . . . </datasetScan>
	<datasetScan location="/root/jg"  path="jg"  name="JGOFFS Archive"   serviceName="OPeNDAP-Server4"> . . . </datasetScan>
	<datasetScan location="/root/ff"  path="ff"  name="FreeForm Archive" serviceName="OPeNDAP-Server4"> . . . </datasetScan>
    Where each <datasetScan> element may have it's own filter and inheritance rules. You MUST NOT lump them all into one <datasetScan> element with one set of filter rules like so: 
    	<datasetScan location="/root"     path="DATA"  name="AllYourDataAreUs"   serviceName="OPeNDAP-Server4"> . . . </datasetScan>
    Because it does not work. If you want them all to be in one a single collection then configure the BES so that it has one top level collection (see Example 1)

  • The path attribute in the <datasetScan> element appears in the URL after the servlet name, and MUST be the same as the value of the location attribute with the leading "/root/" removed. In other words it MUST NOT start with a "/" character .
